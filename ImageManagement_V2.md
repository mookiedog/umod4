## Decision Drivers:

* There is benefit to using image slots as compared to the original giant BSON doc containing all images:

  * Slots can be scanned in a predictable fashion allowing both EP and WP to discover what images are present. If the EP is designed to search its image store at boot it means that EP code updates are decoupled from what images are present in the image store.
  * Images can be updated/added/deleted without having to rewrite the entire image store.

* The image store needs to save complete images instead of just mapblobs: unprotection requires seeing the entire 32K EPROM image, even if one just wants to extract the mapblob

* run-time combining of code and mapblob makes more sense than combining them at build time or image-store load time. The main benefit is for UM4 firmware updates. A new UM4 can be uploaded, and all existing combination builds that reference a "UM4" image will automatically use the new one. Otherwise, every prebuilt image containing UM4 code would need to be updated. If a rider wanted to maintain a specific old UM4 version, they could rename their desired UM4 to include version information, and update the image selector to use the versioned name or the regular name as desired.

  * The only reason to *not* combine images would be if a user wanted to run a non RP58-compatible EPROM. This seems unlikely.
  There is no reason to run a stock RP58-compatible EPROM: a rider would achieve the exact same results by combining that stock EPROM with UM4 code.

* Slot size. The size of the slot seems overkill at 64K bytes, so maybe it is worth figuring out if it can be shrunk. Short answer: not worth the effort.
With 8 megabytes of image_store, that's 127 images which is way too many for a user to manage while on the road.
Even when doing a bunch of tests maybe at a dyno facility, you might need 10 before it all just gets to be too much to manage mentally.

# Image Management V2

The ECU needs to be presented with an EPROM image in order to run.
These EPROM images can be as simple as a stock EPROM .bin files.
More complicated images can combine the ECU code from one EPROM with the entire set of fuel injection maps (the "mapblob") from another EPROM.
Combining the datalogging UM4 firmware with the mapblob from a different EPROM produces a data-logging EPROM featuring the desired mapblob.
For example, combining the UM4 code with the mapblob from an Aprilia Super-D .bin file results in a Super-D datalogger.
For riders who want to explore custom tuning, the UM4 code could be combined with the mapblob from a .bin file generated by a tool such as TunerPro.
The result would be a datalogging version of a user's own full-custom maps.

The tuning process would potentially involve many experiments involving sequences of targeted changes to specific maps.
To make this easier, the umod4 can store over 100 EPROM images in its "image store" flash memory.
Riders can use a phone or a laptop to select which of these EPROM images to run.

## Overview

Image management covers three basic concerns:

1. **Image store management** — adding, removing, renaming, and selecting images stored in EP flash.  Managed via the WP website over WiFi. These images could be stock EPROM .bin files, or they could be EPROM images containing a user's custom mapblob to be combined with UM4 datalogging firmware when the ECU boots.
2. **Image selection** — choosing which stored image the EP loads at boot. Editable from the WP website (on the road) or the PC server (at home).
3. **Image construction** — At boot time, the EP processor assembles the 32KB EPROM image it will present to the ECU from component parts (codebase and an optional mapblob). These components come from the image store.

    *Notes:*
    * *The construction process is trivial: 1 or 2 memory copies.*
    * *Protected EPROMs get 'unprotected' (if possible) just before they take part in the construction process.*

Design philosophy: **EP is kept simple.** WP owns all runtime management. The build produces factory defaults. The PC server handles firmware flashing.

---

## Image Store Layout

The image store holds all the EPROM images available to a rider out on the road.

The image store partition lives at `0x10200000` in EP flash, 8MB total, divided into **128 × 64K "image slots"**.

*Note: The 8MB size is a holdover from previous image store mechanisms.
It could be adjusted to be even larger as the EP uses a 16 MB Flash chip.
That said, the ability to store 128 complete EPROM images is overkill.*

### Slot 0 — Image Selector

The 'Image Selector' is a BSON document describing how the EP will construct the image to be presented to the ECU when the ignition is turned on.

More than one potential image can be described.
The system runs the first image in the array that loads correctly.
Secondary images only get loaded if there is a problem constructing the primary image.

Contains no image data. Up to 64KB. See [Image Selector](#image-selector-slot-0) below.

### Slots 1–127 — Self-Describing Image Slots

Each 64K slot is divided into two equal halves:

| Offset | Size | Content                                                  |
|--------|------|----------------------------------------------------------|
| 0x0000 | 32KB | BSON header: `{name, description, protection, image_m3}` |
| 0x8000 | 32KB | Raw 32KB EPROM image binary                              |

- `name` — stable identifier string, referenced by the selector. Must be unique across slots.
- `description` — human-readable text shown in the WP website UI.
- `protection` — describes the protection mechanism for the EPROM. A value of `'A'` means standard Aprilia daughterboard. A value of `'N'` (or no `protection` key present) means no protection mechanism is required
- `image_m3` — murmur3 hash of the 32KB image data, used for integrity verification.

**Empty slot detection:** an erased flash block is all `0xFF`. The first bytes of a BSON doc are its length, so if the length is 0xFFFFFFFF (4GB), the slot is empty.

---

## Image Selector (Slot 0)

The selector is a BSON document with a single `images` array. Each entry describes how to construct one EPROM image.
Here is an overly complex example specifically to demonstrate various possibilities for generating the image:

```json
{
  "images": [
    {
      "code": "UM4",
      "mapblob": "tuning-experiments-v7",
    },
    {
      "code": "UM4",
      "mapblob": "8796529_Super_D"
    },
    {
      "code": "RP58"
    },
    {
      "code": "UM4"
    }
  ]
}
```

**Field meanings:**

- `code` — name of the slot whose image data provides the codebase (`0x1C00–0x7FFF`) and, if no `mapblob` is specified, the full image.
- `mapblob` *(optional)* — name of the slot whose image data provides the map region (`0x0000–0x1BFF`), overlaid on top of the code slot's maps.
- `continue` *(optional)* — For debug use only: if present, instruct the image loader code to continue on to process the next item in the array list even if the image was constructed successfully.
This allows an image selector to perform a variety of test loads (like validating unprotection mechanisms, etc.) before loading the actual desired image for riding tests.

**Key properties:**

- References are by **name**, not slot number. The selector does not need updating when slots are reorganized.

    *Note that if a user were to use the same name for multiple image slots, the system will only ever find the first instance of that name (lowest slot number).
    It should probably be a warning or error if the same name is reused in the image slot area.*
- The array is a **priority list**. EP tries each entry in order and loads the first one that succeeds.
- The **last entry** should always be a plain `code`-only reference to `"UM4"` — the guaranteed fallback.

In the example above, the EP will try to create an ECU image in the following sequence:

1) Use UM4 datalogging firmware and overlay it with user's own experimental maps
2) Use UM4 datalogging firmware overlaid with Super-D maps, turning the Super-D into a datalogging version of itself
3) Try a plain RP58. This would be identical to inserting an RP58 EPROM into the umod4 socket. No datalogging capability though.
4) Run a plain UM4 build as disaster fallback if none of the previous images load. This represents a datalogging firmware with RP58USA maps.

---

## EP Boot Sequence

1. Read slot 0. If slot 0 is empty (bson length `0xFFFFFFFF`) or fails BSON validation → jump to **limp mode**.
2. For each entry in the `images` array:
   a. Scan slots 1–127, find the slot whose `name` matches `code`. If not found, skip to next entry.
   b. If `mapblob` is specified, scan for that named slot too. If not found, skip to next entry.
   c. Verify `image_m3` from the BSON header against the slot's image data. If mismatch, skip to next entry.
   d. If `protection` key exists and value is "A", read the image data via the descrambler instead of a simple memcpy. This works for entire images or just mapblobs.
   e. If `mapblob` is specified, overlay just the mapblob data onto the previously loaded code image:
      - bytes `0x0000–0x1BFF` come from the mapblob slot's image data
   f. if `continue` key is present, go back to 2 and process the next entry
   g. load the combined 32KB image into the EPROM emulator. Boot complete.
3. If all entries fail → **limp mode** is activated.

### Limp Mode

Limp mode activates a specific 32KB EPROM image compiled directly into EP firmware (not stored in any slot). Default: UM4 code, which contains a 549USA mapblob.

Things must be in a bad state if limp mode ever gets activated.
The image store and/or the image selector must be missing, empty, or corrupted.
As a result, the whole idea behind limp mode is to present the ECU with an image that requires absolutely minimal processing to make it as reliable as possible.

The EP will log the fact that limp mode is active (exact mechanism is TBD).

#### Implementation

The limp mode image is a **raw 32KB array** compiled into EP firmware — no BSON header, no slot format. The limp mode handler is a single `memcpy`.
This is intentional: Putting BSON parsing on this path would just make it more complicated than it needs to be.

The build generates the limp mode image as follows:

```sh
# If a suitable mapblob .bin exists in eprom_lib (e.g. RP58.bin):
head -c 7168 RP58.bin > limp_mode.bin && tail -c +7169 UM4.bin >> limp_mode.bin
# Otherwise fall back to plain UM4 (549USA mapblob):
cp UM4.bin limp_mode.bin

# Convert to C array:
xxd -i limp_mode.bin > limp_mode_image.h
```

CMake checks whether the preferred mapblob `.bin` exists in `eprom_lib/` and selects the appropriate path. The result is a `uint8_t limp_mode_image[32768]` array included directly in EP firmware.

## Map Identification in the Log

The EP will log the names of the code and mapblob making up the image it constructed and runs.
This allows the WP to know what image it is running so that it can present the information on the web status page.
Note that all data logs generated by the UM4 write the *entire* mapblob into the datalog.
This means that a datalog will always be able to indicate exactly what map data it was running when the log was created, even if map names changed or map data was lost over time.

## Build System Changes

### `build-image-store.py` (rewrite)

Input: `.dsc` (JSON metadata) + `.bin` (32KB image) pairs from `eprom_lib/`.

Output per EPROM: a 64K `.slot` file:

- First 32KB: BSON document `{name, description, image_m3}`, zero-padded to 32KB
- Second 32KB: raw image binary

Additional outputs assembled from individual `.slot` files:

- `image_store/image_store.bin` — full 8MB image store binary (slot 0 selector + all image slots), for factory reset flashing

These build outputs are primarily useful for development and testing. In production, a brand-new device with blank flash will boot into limp mode, and the user populates the image store via the WP website. The build-generated `image_store.bin` is therefore a development convenience, not a shipping requirement.

### `memmap_eprom.ld`

- Slot size: 32K → **64K**
- Slot count: 256 → **128**
- Total partition size: unchanged (8MB)

### UF2 Outputs

| File | Contents | Use |
|------|----------|-----|
| `EP.uf2` | Firmware only | Normal firmware updates — image store untouched |

There is no separate `EP_full.uf2`. Factory reset is handled by the flash tool (see [PC Server Changes](#pc-server-changes)).

### Stable Naming Convention

The default UM4 image slot is **always named `"UM4"`**, regardless of firmware version. When a firmware update replaces slot 1 content, the name stays the same. Selectors referencing `"UM4"` continue to work across firmware updates without modification.

## WP Changes

### SWD Flash Writer

The EP reflash flow gains a **"Preserve image store"** option, defaulting to **on**.

- **On (default):** writes `EP.uf2` firmware regions only. Slots 0–127 are untouched.
- **Off (factory reset):** after writing firmware, writes `image_store.bin` to the image store partition (which includes the default selector in slot 0).

After any flash operation: optionally trigger EP reboot via SWD reset so new firmware and/or image store take effect immediately.

### WP Website — Image Store Management Page

A new page (e.g. `eprom_info.html`) provides full image store management over WiFi.

#### Scanning

On page load, WP reads the first few bytes of each 64K slot via SWD. Slots with BSON length `0xFF` are empty. For populated slots, WP reads the full BSON header to get name, description, and hash.

#### List View

Displays all populated slots: name, description, hash (abbreviated), empty/occupied status.

#### Upload New Image

Form fields:

- `.bin` file picker (32KB EPROM image)
- Name (text)
- Description (text)

WP constructs the 64K slot internally (BSON header + image data), writes to the first empty slot via SWD. Uploading an image is independent of image selection — the user separately edits the selector to reference the new image.

*Note: there is no "mapblob only" concept. Every slot holds a complete 32KB EPROM image. A TunerPro user uploads their full TunerPro-generated image. The selector then decides whether to use it as a code source, a mapblob source, or both.*

#### Delete Image

Erases the full 64K flash block for a selected slot via SWD.

#### Rename Image

Rewrites the BSON header portion of a slot (erase + rewrite the full 64K block with updated name).

#### Selector Editor

- Shows current selector entries in priority order
- Each entry: **code EPROM** dropdown (populated from scanned slot names), optional **mapblob EPROM** dropdown
- Actions: reorder entries (drag or up/down), add entry, remove entry
- **Save:** writes updated selector BSON to slot 0 via SWD, then triggers EP reboot

---

## PC Server Changes

- EP firmware flash UI gains a **"Preserve image store"** checkbox (default: checked)
- When unchecked: server writes `image_store.bin` to the image store partition after flashing firmware

## TunerPro User Workflow

1. Generate one or more custom EPROM images in TunerPro → export `.bin` files
2. Connect to umod4 WiFi (AP mode at the track, STA mode at home)
3. Open WP website → Image Store page
4. Upload each `.bin` with a name and description
5. Edit selector: set `code` to `"UM4"` and `mapblob` to the uploaded image's name — EP will splice the custom maps with the current UM4 code at boot
6. Save selector → EP reboots and loads the new image

When UM4 firmware is updated later, no action needed: the selector still references `"UM4"` by name, so EP automatically picks up the new code combined with the stored custom image.

## Transition from Old BSON System

The old BSON partition (`0x10100000`) and `EpromLoader.cpp` are being retired as part of this effort. They are kept alive temporarily only to mine useful code that can be reused in the new image store implementation. Once that code has been extracted, the old BSON partition and loader are removed.

A brand-new device with blank flash will boot into limp mode. There is no fallback to the old BSON system.

## Out of Scope

The following are explicitly deferred and not part of this design:

- PC-side image library manager
- PC-to-EP image sync tool
- Table-level map editing (TunerPro handles this)
- Arbitrary multi-layer profile construction (the old `EpromLoader.cpp` profile system)
- Automated re-upload prompting when UM4 firmware updates
